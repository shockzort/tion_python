# Easy Breezy — требования

Единый документ требований. Заменяет прежние BRD/FRD и `task.md` (консолидация без
избыточности; отброшенное — в §8). Утверждён владельцем 2026-07-11 вместе с планом
переписывания.

Приоритеты: **P0** — MVP (фазы 0–4), **P1** — обязательное после MVP (фазы 5–7),
**P2** — желательное (фаза 8+).

## 1. Назначение

Локальный клиент-серверный сервис управления бризерами Tion по BLE — без облака Tion
и нестабильной базовой станции MagicAir. Голосовое управление через Умный дом Яндекса,
веб-PWA как единый интерфейс (браузер + телефон), автоматизация по времени и датчикам,
задел под локальную LLM.

## 2. Окружение и оборудование

| | |
|---|---|
| Сервер | Linux (Raspberry Pi 4 / Orange Pi 5 / x86), Python 3.12, BLE-адаптер (Bluetooth ≥ 4.2, BlueZ) |
| Бризеры | 3 × Tion S4 — все в радиусе BLE одного сервера |
| Датчики CO₂ | Tion CO2+ / MagicAir — данные через облачный API MagicAir (P1); BLE-реверс датчиков — отдельный R&D-трек вне этого документа |
| Сторонние датчики | температура/влажность/CO₂ по MQTT (P1) |
| Внешний доступ | VPS с frp-туннелем (белый IP ненадёжен), nginx + TLS |
| Клиенты | браузеры Chrome/Firefox/Safari, Android-телефон (PWA) |

## 3. Функциональные требования

### 3.1. Управление бризером (P0)

- FR-1. Вкл/выкл, скорость 1–6, нагрев вкл/выкл, целевая температура 10–30 °C,
  режим приток/рециркуляция, звук вкл/выкл, подсветка вкл/выкл.
- FR-2. Чтение состояния: всё из FR-1 + температура входящего воздуха и воздуха после
  бризера, остаток ресурса фильтра (дни), код ошибки, доступность устройства.
- FR-3. Состояние отдаётся из кэша сервера (последнее подтверждённое) с меткой
  свежести; живой BLE-запрос в пути чтения запрещён.
- FR-4. Команды идемпотентны: повтор с тем же ключом идемпотентности не приводит к
  повторному исполнению и возвращает тот же результат (защита от ретраев Алисы и
  повторных нажатий).
- FR-5. Команды на одно устройство сериализуются; конкурирующие изменения не теряют
  поля (запись в S4 — полный набор параметров).

### 3.2. Управление парком устройств (P0)

- FR-6. Поиск устройств BLE-сканом с фильтром по имени, отображение RSSI.
- FR-7. Сопряжение из UI (мастер с инструкцией; физически BLE выполняет сервер),
  отвязка (unpair + удаление bond), удаление (мягкое — история сохраняется),
  переименование, назначение комнаты.
- FR-8. Группы устройств; групповые команды («выключи все бризеры»).
- FR-9. CLI-инструмент для тех же операций (диагностика без UI).

### 3.3. Голос: Умный дом Яндекса (P0)

- FR-10. Устройства видны в приложении «Дом с Алисой»; поддержаны фразы уровня:
  «включи бризер в спальне», «поставь скорость три», «включи обогрев»,
  «сделай 22 градуса», «выключи звук на бризере».
- FR-11. Запрос статуса голосом («какая температура на бризере…»).
- FR-12. Привязка аккаунта через наш OAuth2-провайдер; отвязка по `unlink`.
- FR-13. Изменения состояния отправляются в Яндекс push-колбэком (состояние видно в
  приложении Дома без опроса).
- FR-14. Недоступное устройство → корректный код ошибки Яндексу (DEVICE_UNREACHABLE).

### 3.4. Веб-PWA (P0 — дашборд/управление/сопряжение; P1 — графики)

- FR-15. Русскоязычный адаптивный интерфейс; устанавливается как PWA на Android;
  обновляется с сервера без переустановки.
- FR-16. Дашборд: карточки устройств с состоянием и управлением (FR-1), бейджи
  «нет связи» / «ручной режим», плитки сценариев, «выключить все».
- FR-17. Обновление состояния в реальном времени (WebSocket), optimistic-обновления
  с откатом при ошибке команды.
- FR-18. Экраны: устройства (мастер сопряжения, комнаты, группы), автоматизация,
  графики, настройки.
- FR-19. Локальный доступ по LAN и удалённый через HTTPS (frp) с авторизацией.

### 3.5. Автоматизация (P1)

- FR-20. **Сценарий** — именованный список действий (устройство/группа → набор
  параметров); запуск кнопкой, расписанием, триггером, интентом.
- FR-21. **Расписание** — cron-выражение (локальная TZ) → сценарий или действия.
  Пример: «ночной режим в 23:00 ежедневно».
- FR-22. **Триггер** — условие по метрике датчика с порогом, **гистерезисом**,
  кулдауном и временным окном; действия на входе и выходе из состояния.
  Примеры: CO₂ > 1000 ppm → турбо, CO₂ < 800 → обычный; T > 25 °C → нагрев выкл.
- FR-23. Ручное управление (UI/Алиса/CLI/интент) ставит устройству **manual-hold**
  (по умолчанию 60 мин, настраивается): автоматика в этот период пропускается
  (журналируется), UI показывает бейдж и кнопку «вернуть автоматику».
- FR-24. CRUD всего перечисленного в UI; хранение в БД; выполнение журналируется.

### 3.6. Датчики (P1)

- FR-25. MagicAir cloud: опрос датчиков CO2+/MagicAir (CO₂, температура, влажность)
  раз в 60 с; маркировка устаревших данных (>5 мин).
- FR-26. MQTT: подписка на топики сторонних датчиков (число или JSON), реестр
  датчиков в UI; недоступность брокера не валит сервис.
- FR-27. Датчики доступны в триггерах, телеметрии и экспонируются в Алису
  (сенсоры климата: CO₂/температура/влажность).

### 3.7. Телеметрия (P1)

- FR-28. Запись истории: по устройствам — температуры, скорость, нагрев, фильтр;
  по датчикам — все метрики. Сырьё 7 дней, часовые агрегаты (min/max/avg) 2 года.
- FR-29. Графики за период в UI; API выборки диапазона.

### 3.8. Текстовые интенты — задел под LLM (P1 интерфейс+правила, P2 LLM)

- FR-30. `POST /api/intents/execute {text}` → разбор в интент → командная шина →
  человекочитаемый ответ. Провайдер разбора — сменный: v1 rule-based русский парсер
  (действия, числительные, имена комнат/устройств/сценариев), v2 — OpenAI-совместимый
  локальный LLM (function calling) под тем же контрактом.
- FR-31. Низкая уверенность разбора → ответ-уточнение, а не действие.

### 3.9. Уведомления (P2)

- FR-32. Web push (VAPID): бризер недоступен >10 мин, провал бэкапа, критические
  ошибки. Подписка из PWA.

## 4. Нефункциональные требования

- NFR-1. **Латентность**: голосовая команда → действие на устройстве < 3 с (бюджет
  Яндекса); ответ на query — из кэша, < 300 мс на сервере.
- NFR-2. **Доставка команд**: at-least-once от источников, exactly-once эффект через
  дедупликацию по ключу идемпотентности; журнал команд с итогами (30 дней).
- NFR-3. **Персистентность**: устройства, группы, сценарии/расписания/триггеры,
  датчики, телеметрия, пользователи/сессии/токены, привязка Яндекса — SQLite (WAL),
  только асинхронный доступ, миграции Alembic. Рестарт сервиса не теряет ничего.
- NFR-4. **Отказоустойчивость**: автопереподключение BLE с экспоненциальной задержкой
  1→60 с; таймауты на всех I/O; деградация с честными статусами; watchdog systemd;
  все устройства офлайн >10 мин → перезапуск BLE-адаптера, затем сервиса.
- NFR-5. **Безопасность**: пароль администратора — argon2id; сессии/токены — только
  хэши в БД; секреты в `.env` (в git не попадают); внешний доступ только HTTPS;
  API закрыт авторизацией; троттлинг логина.
- NFR-6. **Логи**: structlog JSON с ротацией; BLE-события — отдельный файл; уровни
  по назначению (DEBUG BLE-детали, INFO события, WARNING нештатное, ERROR сбои).
- NFR-7. **Бэкапы**: ежедневный снапшот БД (`VACUUM INTO` + gzip), хранить 7,
  проверенное восстановление.
- NFR-8. **Развёртывание**: bare systemd (основной путь) и docker-compose
  (альтернатива); установка по runbook; конфигурация через `.env`.
- NFR-9. **Качество**: ruff/black/isort, mypy strict; pytest, покрытие ≥ 80 %;
  тесты не требуют BLE-железа и сети; CI на GitHub Actions.
- NFR-10. **Расширяемость**: новые модели бризеров — интерфейс драйвера; новые
  источники датчиков — интерфейс источника; новые голосовые провайдеры — интерфейс
  интентов; транспорт BLE абстрагирован (задел под BLE-прокси).
- NFR-11. **Ресурсы**: целевой RSS сервера < 300 МБ на RPi4; сборка фронтенда вне Pi.

## 5. Утверждённые решения

| Вопрос | Решение |
|---|---|
| Модели | только S4 (3 шт.); S3/Lite — интерфейс есть, драйверы по потребности |
| Алиса | платформа «Умный дом» (не диалоговый навык) |
| CO₂ | MagicAir cloud сначала, BLE-реверс — R&D отдельно |
| Внешний доступ | frp через VPS; TLS: LE IP-сертификат (эксперимент) или бесплатный поддомен DuckDNS — решение в фазе 4 |
| Мобильный клиент | PWA; Capacitor APK — опционально позже; телефон BLE не делает |
| LLM | интерфейс интентов + rule-based сейчас, локальная LLM позже |
| Хранилище | SQLite + SQLAlchemy async + Alembic; без Redis |
| Планировщик | croniter + свой цикл (инжектируемые часы) |
| OAuth-провайдер | свой минимальный (authorization_code + refresh) |
| Реалтайм UI | WebSocket |
| Лицензия | приватный проект, без лицензии |
| Имя | Easy Breezy (`easy_breezy`, `easy-breezy.service`, CLI `breezy`) |

## 6. Ограничения и допущения

- Радиус BLE ~10 м с препятствиями; все устройства в зоне действия сервера.
- Один пользователь-администратор (домашний сервис); многопользовательность не
  требуется, но API-токены позволяют интеграции.
- Данные CO₂ через облако требуют работающей базовой станции и интернета — деградация:
  триггеры по CO₂ замирают с предупреждением, остальной сервис не страдает.
- Сервер требует постоянного питания; частичная работа UI офлайн (app-shell), данные
  и команды — только при доступности сервера.

## 7. Приёмка MVP (конец фазы 4)

1. С телефона вне дома через PWA: вкл/выкл бризера, скорость, нагрев.
2. «Алиса, включи бризер в спальне», «поставь скорость три», «включи обогрев»,
   «сделай 22 градуса» — исполняются < 3 с; состояние в «Доме с Алисой» соответствует
   реальному.
3. Повтор команды Алисой (ретрай) не вызывает повторного исполнения.
4. Обесточенный бризер: Алиса — «недоступно», UI — бейдж; после включения питания
   реконнект < 90 с без рестарта сервиса.
5. Рестарт сервера: устройства, сессии, привязка Яндекса — на месте.
6. Сопряжение нового бризера из UI-мастера без CLI.

## 8. Отброшено из прежних BRD/FRD (с причинами)

| Требование | Причина |
|---|---|
| Redis-кеш | лишняя зависимость: state cache в процессе, SQLite хватает |
| «Шифрование данных аутентификации AES-256» | заменено корректной практикой: argon2id-хэши, TLS, секреты в `.env` |
| Email-уведомления | заменены web push (PWA); email-инфраструктура не нужна |
| CI/CD с автодеплоем | домашний сервис: CI обязателен, деплой — по runbook/Makefile |
| «До 10 одновременных пользователей» | не метрика домашнего сервиса; фактическая ёмкость выше |
| Диалоговый навык Яндекса | выбран Умный дом — нативные фразы без «попроси навык…» |
| Поддержка S3/Lite в MVP | устройств нет; протокол задокументирован, драйверы — по потребности |
| «Регулярное обновление TLS-сертификатов» как процедура | автоматизировано certbot'ом — не требование, а деталь развёртывания |
| MQTT как обязательный компонент MVP | перенесено в P1: единственный источник CO₂ на MVP не нужен |
