# ADR-0007. Вечные задачи не имеют права умирать молча

Дата: 2026-07-12. Статус: принято (полевой инцидент отладочного стенда MVP).

## Контекст

Сервис держится на долгоживущих asyncio-задачах: супервизоры устройств,
воркеры командной шины, циклы телеметрии, нотификатор Яндекса. Инцидент:
`DriverError` («соединение разорвано» во время первого чтения) не входил в
except-список цикла супервизора — задача умерла молча, устройство навсегда
зависло в «connecting», в логе — ни одной записи за час. Незавершённая
задача с исключением не печатает traceback, пока жива ссылка на неё
(`Task exception was never retrieved` появляется только при сборке мусора).

## Решение

Каждая вечная задача обязана строиться по шаблону:

1. Ожидаемые сбои итерации ловятся типизированно и обрабатываются политикой
   (ретрай с backoff, пропуск итерации) с логом WARNING.
2. `asyncio.CancelledError` пробрасывается — отмена остаётся штатным
   способом остановки.
3. Всё остальное ловится страховкой `except Exception` c `log.exception`
   и продолжением цикла — деградация видимая, но живая.

Точки применения: `DeviceSupervisor.run`, воркеры `CommandBus`, циклы
`TelemetryService`, `YandexNotifier`; каждая новая вечная задача — по тому же
шаблону. Ревью-чек: `create_task` без владельца, который её await'ит или
обрабатывает исключения, — запах.

## Последствия

- Сбой любой природы стоит одной итерации цикла, а не подсистемы до рестарта.
- `log.exception` в вечном цикле — сигнал завести типизированную обработку;
  страховка не замена явным исключениям, а последний рубеж.
- Голый `except Exception` с re-raise/логированием разрешён CLAUDE.md именно
  для этого случая.
