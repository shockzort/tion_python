# ADR-0005. Сосуществование с другими BLE-центральными

Дата: 2026-07-11. Статус: принято (Фаза 1, полевые факты — спека §5/§1.6).

## Контекст

BLE-периферия держит одно соединение. За бризеры конкурируют: приложение
Tion Remote (держит соединения, пока открыто), станция MagicAir
(периодически подключается и приводит mode/heater/temp к своей политике,
меняет fan по CO₂). Занятый бризер не рекламируется: «не найден в эфире»
не отличим от «выключен» без контекста.

## Решение

- Сервис удерживает постоянные соединения через супервизор (монополия на
  бризер, пока сервис жив); разрывы лечатся backoff-реконнектом.
- «Не найден в эфире» трактуется как временная занятость/недоступность —
  ретраи без паники; стабильный HCI 0x3E — признак «не сопряжён» (ADR-0003),
  эскалация пользователю.
- Ручные изменения защищаются manual-hold окнами command bus (Фаза 2):
  на время hold сервис перезаписывает чужие откаты.
- MagicAir остаётся в системе на переходный период; владелец выводит его
  после запуска сервиса (CO₂-логика переезжает в наши сценарии, Фаза 6).

## Последствия

- Пока сервис держит соединение, Tion Remote и MagicAir к бризеру не
  подключатся — фактическое поведение системы после запуска сервиса.
- Смоуки и отладка с работающим MagicAir должны учитывать его откаты
  (это не баги кодека).
