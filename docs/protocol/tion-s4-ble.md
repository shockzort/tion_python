# BLE-протокол бризеров Tion — семейство Lite (S4, Lite) и S3

Спецификация для чистой (clean-room) реализации. Источники знаний:

- реверс-инжиниринг публичного проекта `TionAPI/tion_python` (протокольные **факты**: UUID, опкоды, форматы — не являются объектом авторского права);
- собственный анализ захваченных кадров (golden-векторы в `server/tests/golden/`);
- алгоритм CRC определён нами экспериментально по двум независимым захватам.

Каждый факт имеет статус: **[✓]** — проверен на захваченных кадрах; **[u]** — наблюдение
апстрима, не перепроверено; **[?]** — гипотеза, требует проверки на железе (Фаза 1).

Термины: *пакет* — единица BLE-записи/нотификации (≤20 байт); *кадр* — логическое
сообщение, собранное из пакетов. Все многобайтовые числа — little-endian, если не
указано иное. Температуры — знаковый int8 (двоичный дополнительный код). **[✓]**

## 1. Семейство Lite (Tion S4, Tion Lite)

### 1.1. Обнаружение и GATT

- Имя S4 в эфире: **`Breezer 4S`** (без «Tion»; проверено сканом на трёх
  устройствах, 2026-07-11) **[✓]**. Апстрим упоминал также `Tion Breezer` /
  `tion_` (возможно, другие модели/прошивки) **[u]** — фильтр сканера учитывает
  оба варианта. Модель различается по подстроке (`4S`, `Lite`, `3S`).
- Происхождение UUID: реверс апстрима (`tion_btle/light_family.py` в git-истории);
  из захватов кадров UUID непроверяемы (в кадрах их нет), поэтому статус всех
  UUID — [u] до перечисления GATT-таблицы реального S4 (§4, п. 1). Косвенные
  подтверждения: S3 использует стандартный Nordic UART Service, CCCD `0x2902` —
  стандарт Bluetooth SIG, те же UUID — в независимых реализациях (ESPHome-Tion).
- Сервис: `98f00001-3788-83ea-453e-f52244709ddb` — **[✓]** подтверждён на живом
  железе (advertisement трёх S4 + GATT-перечисление после бонда, 2026-07-11); адрес
  устройства типа **random static**, AdvertisingFlags `0x04` (BR/EDR not supported —
  чистый LE).
- Manufacturer data company `0xffff`, payload `<mac 6 байт> 03 80 00 00 <pair>`:
  последний байт `00` — обычный режим, `01` — **режим сопряжения** (флип пойман
  живьём в момент нажатия кнопки, 2026-07-11). **[✓]** Семантика `03 80` не раскрыта. **[?]**
- Характеристика записи: `98f00002-3788-83ea-453e-f52244709ddb` **[✓]** (живой обмен:
  запрос состояния записан и принят прошивкой, 2026-07-11).
- Характеристика нотификаций: `98f00003-3788-83ea-453e-f52244709ddb` (CCCD `0x2902`) **[✓]**
  (подписка выполнена, кадр состояния получен нотификациями, 2026-07-11).
- Общение: клиент пишет кадры-запросы в write-характеристику, устройство отвечает
  кадрами через notify. **[✓]** (get-обмен; поведение после SET — см. §4 п. 2).

### 1.2. Транспортное кадрирование

Кадр длиннее 19 байт режется на пакеты. Первый байт каждого пакета — тип:

| Тип | Значение | Семантика |
|---|---|---|
| `0x80` | SINGLE | кадр целиком в одном пакете |
| `0x00` | FIRST | первый пакет многопакетного кадра |
| `0x40` | MIDDLE | продолжение |
| `0xc0` | END | последний пакет |

Сборка: конкатенация содержимого пакетов **без** типовых байтов. **[✓]**
Разрезка при отправке: полезные чанки по 19 байт + типовой байт. **[✓u]**
Пакет с неожиданным типом (MIDDLE/END без FIRST) — сброс сборки, лог. Новый FIRST/SINGLE
всегда начинает сборку заново.

### 1.3. Структура кадра

```
смещение  размер  поле
0         2       length     — длина кадра в байтах (включая это поле и CRC)   [✓]
2         1       magic      — константа 0x3a                                   [✓]
3         1       rand       — произвольный байт (апстрим: случайный; в getStatus
                               S4 константа 0xa1 — семантика не влияет)          [u]
4         2       opcode     — код команды/ответа                               [✓]
6         4       request_id — идентификатор запроса; ответ ЭХОМ возвращает
                               request_id запроса (подтверждено фикстурой Lite)  [✓]
10        4       extra      — назначение неизвестно (в запросах апстрим слал
                               случайные байты; в ответах — некий счётчик)       [?]
14        N       payload    — полезная нагрузка (зависит от opcode)
14+N      2       crc        — CRC-16/CCITT-FALSE, big-endian                    [✓]
```

**CRC** — CRC-16/CCITT-FALSE (poly `0x1021`, init `0xffff`, без отражений, xorout 0),
считается по кадру от поля `length` до конца `payload` (т.е. по всему кадру без
2 байт CRC; транспортный типовой байт не участвует), записывается **big-endian**.
Определено экспериментально, подтверждено на обоих захватах (Lite `b5 ad`,
S4 `98 5d`). **[✓]**

Примечание: апстрим не знал алгоритма и слал в запросах константу `bb aa` —
устройства такие запросы принимают **[u]**. Наша реализация отправляет корректный
CRC и проверяет CRC входящих кадров в мягком режиме (несовпадение — warning в лог,
кадр не отбрасывается) до полевого подтверждения на всех прошивках; после
подтверждения — строгий режим.

Корреляция: сопоставление ответа запросу — по `request_id` (эхо). Устройство может
присылать кадры состояния и без запроса (push при изменениях **[?]**) — обработчик
обязан принимать кадры с неизвестным `request_id`.

### 1.4. Опкоды

| Опкод (hex, wire-порядок) | Направление | Назначение |
|---|---|---|
| S4 `32 32`, Lite `32 12` | → | REQUEST_PARAMS — запрос состояния **[✓u]** |
| S4 `31 32`, Lite `31 12` | ← | ответ состояния (наблюдение: первый байт опкода ответа = первый байт запроса − 1; подтверждено обоими захватами) **[✓]** |
| S4 `30 32`, Lite `30 12` | → | SET_PARAMS — установка параметров **[u]** |
| S4 `32 33`, Lite `09 40` | → | REQUEST_DEVICE_INFO — формат ответа не задокументирован **[?]** |

REQUEST_PARAMS и REQUEST_DEVICE_INFO отправляются с пустым payload. **[u]**

### 1.5. Ответ состояния S4 — payload (наблюдалась длина 31 байт)

| Смещение | Поле | Примечание |
|---|---|---|
| 0 | флаги | бит0 power, бит1 sound, бит2 light, **бит4 heater — ИНВЕРТИРОВАН** (0 = нагрев вкл) **[✓]**; биты 3,5,6,7 — неизвестны **[?]** |
| 1 | ? | в захвате `0x51` **[?]** |
| 2 | mode | 0 = приток (outside), 1 = рециркуляция **[✓]** (запись/чтение живьём 2026-07-11: mode=1 принят и виден в перечитанном состоянии; физику заслонки на слух не проверяли; MagicAir откатывает — §1.6) |
| 3 | heater_temp | целевая температура нагрева, °C **[✓]** |
| 4 | fan_speed | скорость 1–6 **[✓]** |
| 5 | in_temp | температура входящего воздуха, int8 °C **[✓]** |
| 6 | out_temp | температура после бризера, int8 °C **[✓]** |
| 7–16 | ? | в захвате `1b 26 3b 6e 07 00 fa 4e 07 00`; байты 13–16 (`fa 4e 07 00` LE = 479 994 с ≈ 5.6 сут) похожи на счётчик времени — кандидат electronic_work_time по аналогии с Lite **[?]** |
| 17–19 | filter_remain | остаток ресурса фильтра, **секунды**, uint24 LE (в захвате 15 073 030 с ≈ 174.5 сут) **[✓]** |
| 20 | ? | `0x00` **[?]** |
| 21–23 | ? | `a6 e9 22` LE = 2 288 038 с ≈ 26.5 сут — кандидат device_work_time **[?]** |
| 24–30 | ? | `00 00 00 00 00 06 00`; байт 29 (`0x06`) — кандидат error_code/счётчик ошибок (у Lite error_code = payload[28]) **[?]** |

Разбор обязан быть терпим к длине: читать известные поля по смещениям, длину
payload не фиксировать жёстко (разные прошивки могут расширять хвост).

### 1.6. SET_PARAMS S4 — payload запроса (7 байт)

```
[флаги][0x00][mode][heater_temp][fan_speed][sign:2 LE]
```

- Флаги записи: бит0 power, бит1 sound, бит2 light, **бит3 heater — ИНВЕРТИРОВАН**
  (0 = нагрев вкл), **бит4 — константа 1**. Обратите внимание на асимметрию с чтением:
  при чтении heater — бит4, при записи — бит3. **[u]**
- Биты 5–7 по комментарию апстрима — resetSettings / resetErrorCounter /
  resetFilterResource; не проверялись, наша реализация всегда шлёт 0. **[?]**
- `sign` — константа 181 (`b5 00`); назначение неизвестно, апстрим слал всегда. **[u]**
- SET_PARAMS задаёт **полный набор параметров разом** (не дельту) — отправитель обязан
  заполнить все поля актуальными значениями. Следствие для сервиса: записи на одно
  устройство сериализуются, дельта мержится поверх последнего подтверждённого
  состояния. **[✓]** (живой обмен 2026-07-11)
- Подтверждение: устройство присылает кадр состояния сразу после применения SET,
  без явного запроса (~150 мс). **[✓]** Прошивка вправе **скорректировать** запрошенные
  поля (наблюдение: включение рециркуляции сопровождалось fan 3→2 в подтверждении). **[?]**
- Стратегия реализации: ждать кадр после SET (таймаут 3 с), затем **всегда**
  контрольный REQUEST_PARAMS — кадр подтверждения может потеряться при сборке,
  а поля могут быть скорректированы; истина — только перечитанное состояние. **[✓]**
- **Внешние контроллеры перезаписывают состояние**: MagicAir с CO₂-автоуправлением
  в течение ~минут откатывает mode/heater/heater_temp к своей политике и меняет
  fan_speed по CO₂ (полевой факт: наши mode=recirculation и heater=on откатились
  за 1–3 мин; fan самовольно 2→4). «SET откатился» в смоуке — это не баг кодека.
  Следствие для сервиса: manual-hold окна (план §8) и/или отключение
  CO₂-автоуправления на время ручного управления. **[✓]**

### 1.7. Пейринг S4

Только сопряжение на уровне ОС (BlueZ bonding), устройство должно находиться в
режиме сопряжения (кнопка на корпусе ~5 с до синего мигания; надёжнее всего
триггерится из выключенного состояния — выключить, затем зажать). Прикладной команды
пейринга у семейства Lite нет (в отличие от S3). Проверено живьём 2026-07-11: **[✓]**

- До сопряжения бризер молча игнорирует LL-соединения незнакомых центральных (§5) —
  без режима сопряжения первое подключение невозможно в принципе. **[✓]**
- Рабочий рецепт: BlueZ `Device1.Pair()` на свежем объекте устройства (скан →
  сразу Pair, агент NoInputNoOutput, adapter `Pairable: yes`) → `Paired=Bonded=true`
  мгновенно. **[✓]**
- Порядок критичен: **SMP до GATT**. В режиме сопряжения бризер принимает
  LL-соединение и без бонда, но рвёт его через ~2.8 с, если центральный начал
  GATT-discovery вместо сопряжения (поэтому `connect()`+`pair()` поверх готового
  соединения — как в bleak — не работает). **[✓]**
- `Pair()` вне режима сопряжения отбивается ложной ошибкой
  `ConnectionAttemptFailed: Page Timeout` (BR/EDR-код на чистом LE-устройстве;
  это «LL-игнор» в обёртке BlueZ, а не классический пейджинг). Если SMP начался,
  но на адаптере `Pairable: no` / нет зарегистрированного агента —
  `AuthenticationFailed`. `breezy pair` самодостаточен: на время пейринга сам
  регистрирует JustWorks-агент (`org.bluez.Agent1`, NoInputNoOutput) и включает
  `Pairable`, после — возвращает как было. Сопряжены все три бризера (EC, EB, D0):
  паттерн «Page Timeout ×N до кнопки → мгновенный успех» воспроизведён трижды. **[✓]**
- Бонд снимает блок: обычный connect + GATT-обмен работают без кнопки. **[✓]**
  Персистентность бонда после power-cycle бризера — проверить (§4 п. 0). **[?]**

### 1.8. Отличия Tion Lite от S4

- Опкоды: см. §1.4.
- Payload ответа состояния (наблюдалась длина 57 байт):
  - байт 0, флаги: бит0 power, бит1 sound, бит2 light, бит4 filter_change_required,
    бит5 co2_auto_control (признак внешнего управления от MagicAir), **бит6 heater —
    НЕ инвертирован**, бит7 have_heater **[✓u]**;
  - байт 2 mode: **0 = рециркуляция, 1 = смешанный, ≥2 = приток** (нумерация другая!) **[✓u]**;
  - байты 3–6 — как у S4 (heater_temp, fan_speed... точнее: 3 heater_temp, 4 fan_speed,
    5 in_temp, 6 out_temp) **[✓]**; байт 7 electronic_temp **[u]**;
  - 8–10 electronic_work_time uint24 LE, сек **[✓]**; 16–19 filter_remain uint32 LE,
    сек **[✓]**; 20–23 device_work_time uint32 LE, сек **[✓]**; 28 error_code **[✓]**.
- SET_PARAMS Lite — payload 15 байт:
  `[флаги][0x00][tb][heater_temp][fan_speed][presets:6][0x60 0x00][0x00]`,
  флаги: бит0 power, бит1 sound, бит2 light, **бит4 heater (НЕ инвертирован, позиция
  чтения = позиции записи)**; `tb` = 0x02 если heater_temp>0 или fan_speed>0, иначе
  0x01; `presets` — константа апстрима `0a 14 19 02 04 06`. Семантика tb/presets не
  раскрыта. **[u]**

## 2. Tion S3 (Nordic UART, справочно — драйвер вне текущего скоупа)

- GATT: сервис `6e400001-b5a3-f393-e0a9-e50e24dcca9e`, write `6e400002-…`,
  notify `6e400003-…` (Nordic UART Service). **[u]**
- Кадры **фиксированные 20 байт**, без сборки и без CRC.
- Запрос: `[0x3d][command][special][0x00 × 16][0x5a]`; special=1 только для PAIR.
  Команды: 1 = REQUEST_PARAMS, 2 = SET_PARAMS, 5 = PAIR. **[u]**
- Ответ начинается `0xb3`, заканчивается `0x5a` **[✓]** (захват в golden):

| Байт | Поле |
|---|---|
| 2 | низкий ниббл fan_speed, высокий ниббл mode (0 рецирк., 1 смеш., ≥2 приток) |
| 3 | heater_temp |
| 4 | бит0 heater, бит1 power, бит2 timer, бит3 sound |
| 7 / 8 | out_temp / in_temp (int8) |
| 9–10 | filter_remain, **дни**, uint16 LE |
| 11–12 | время устройства чч:мм |
| 13 | error_code |
| 14 | производительность, м³/ч |
| 17–18 | версия прошивки (LE, отображать как hex) |

- Запись SET_PARAMS: байт2 fan_speed, байт3 heater_temp, байт4 mode,
  байт5 = heater | power<<1 | sound<<3. **[u]**
- Пейринг S3 — **прикладной**: после BlueZ bonding отправить кадр PAIR (команда 5,
  special=1). **[u]**

## 5. Наблюдения полевого смоука (2026-07-11, ноутбук с BlueZ 5.85, Intel AX211)

- Имя `Breezer 4S`, сервисный UUID подтверждён (§1.1). Три S4 видны стабильно.
- Manufacturer data (company `0xffff`): `<mac LE> 03 80 00 00 00` — у всех трёх
  бризеров одинаково (вне режима сопряжения). Гипотеза: байты после MAC кодируют
  статус сопряжения — сверить в режиме сопряжения. **[?]**
- **До сопряжения бризер не принимает LE-соединение: молча игнорирует `CONNECT_IND`
  незнакомого центрального.** Сигнатура на HCI (btmon): `LE Enhanced Connection
  Complete: Success` → ровно через 6 conn-интервалов (~245 мс при 45 мс) `Disconnect:
  Connection Failed to be Established (0x3E)` — от периферии не пришло ни одного
  пакета. Детерминированно (0/30+ суммарно, оба ближних бризера), RSSI −70,
  инициация LE 1M, own address Public, простаивающий адаптер. **[✓]**
- **Адаптер исправен** — контрольный эксперимент: тем же адаптером в той же серии
  подключились **3/3 чужих LE-устройства** (в т.ч. два по random-адресам, GATT-таблицы
  прочитаны), бризер — 0/2. Ранние версии причин — «активный A2DP забивает радио»,
  «WiFi coexistence combo-чипа», «неисправный BT-модуль» — **опровергнуты**: сбой
  детерминированно воспроизводится на полностью свободном радио (без A2DP/HID),
  WiFi при этом на 5 ГГц. **[✓]**
- Единичный успех connect в ранней серии (~1/20 при активном A2DP) объяснения не
  имеет (возможно, окно после power-cycle или ложная интерпретация) — на выводы
  не влияет. **[?]**
- Устаревшая запись в кэше BlueZ (`Trusted: yes` при `Paired: no`) давала ложный
  `org.bluez.Error.ConnectionAttemptFailed: Page Timeout` (код классического BR/EDR).
  `bluetoothctl remove <mac>` убирает ложную ошибку → остаётся чистый LE-таймаут.
- **Сопряжение выполнено живьём** (EC:82:9F:A4:90:14): кнопка ~5 с → в рекламе
  последний байт mfr data `00`→`01` → `Device1.Pair()` → `Paired=Bonded=true` →
  обычный `breezy state` без кнопки прочитал корректное состояние (все поля
  правдоподобны). Детали и порядок SMP-до-GATT — §1.7. **[✓]**
- **SET-смоук пройден** (fan, heater+temp, mode): изменения применяются, подтверждение
  приходит само (~150 мс, §1.6), латентность «запись→кадр» ~0.15 с (цель < 2 с — запас
  ×13). Нагрев проверен физикой: out_temp 22→24 °C за минуту работы. Инверсия бита
  heater при записи корректна (§1.6). CRC наших запросов прошивка принимает; ответы
  без `crc_mismatch` — CRC подтверждён обоюдно (§4 п. 3–4). **[✓]**
- Разовые сбои сборки подтверждения / разрывы в момент SET наблюдались (флаки BLE
  и/или конкуренция MagicAir) — лечатся контрольным REQUEST_PARAMS (§1.6) и ретраем;
  на устойчивость чтения не влияют. **[✓]**
- **Бризер не рекламируется, пока кем-то удержано соединение** (наблюдение: все три
  разом пропали из эфира — вероятно, открытое приложение Tion на телефоне; вернулись
  по одному). BLE-периферия допускает одного центрального: наш сервис будет
  конкурировать за соединение с приложением Tion/MagicAir. Следствия: «не найден
  в эфире» ≠ «выключен» (возможно «занят другим центральным»); супервизору не
  паниковать на серию пустых сканов. **[✓]**
- Следствие для реализации: первому обращению к бризеру обязательно предшествуют
  режим сопряжения + BlueZ bonding (§1.7) — это разовое действие на каждый бризер
  с каждого хоста. Супервизор должен отличать «не сопряжён» (стабильный 0x3E —
  ретраи бессмысленны, нужна эскалация пользователю) от временной недоступности.
  Активный скан в момент connect избегать (наш транспорт так делает).

## 4. План уточнений на железе (Фаза 1, чек-лист)

0. **Сопряжение** (приоритет смоука, см. §5): подтвердить, что в режиме сопряжения
   LL-соединение устанавливается и бонд снимает блок навсегда (переживает
   power-cycle бризера и перезагрузку хоста); зафиксировать поведение manufacturer
   data в режиме сопряжения; проверить `breezy pair`/`unpair` end-to-end.
1. Формат имени S4 в эфире (скан) и GATT-таблица: перечислить сервисы/характеристики,
   подтвердить UUID из §1.1 (`bluetoothctl` → `menu gatt` → `list-attributes`).
2. Поведение после SET: приходит ли кадр состояния без запроса (§1.6).
3. Приём запросов с корректным CRC (ожидаемо да — принимает даже мусорный трейлер).
4. Строгая валидация CRC ответов на реальной прошивке → перевод в строгий режим.
5. Кандидаты неизвестных полей payload (§1.5): сверка со значениями в приложении Tion
   (наработка, ошибки), фиксация в этом документе.
6. REQUEST_DEVICE_INFO: снять ответ, задокументировать формат (серийник/прошивка?).
