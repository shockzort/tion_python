## Пост-фазные работы

### 2026-07-12 — Фаза 8: разведка CO2+ закрыла прямой BLE; принят вариант A

R&D на боевом NUC показал: у модуля CO2+ (SP320) нет BLE-интерфейса для
прямого подключения — связь со станцией по RF-радиоканалу (вероятно
868 МГц), инициализация через базовую станцию, в BLE-эфире модуль не
виден (метод, факты, ссылки — `docs/protocol/tion-co2plus-ble.md`).
Исходная постановка «CO2+ как бризер по BLE» неисполнима.

**Решение владельца (2026-07-12): вариант A** — источник CO2+ остаётся
облачным MagicAir (реализовано в Фазе 6, работает). Новой разработки по
Фазе 8 нет, фаза закрыта. Локальный BLE-датчик не от Tion (вариант B)
уходит в бэклог на случай отказа от облака. Осталось владельцу — вернуть
модуль в приложение MagicAir (при разведке был штатно отвязан от
станции). Урок: контрольный эксперимент против гипотезы до написания
кода (ср. история с адаптером 0x3E).

**Итог проекта:** все 9 фаз закрыты, MVP функционально завершён и
работает на боевом сервере (NUC, v0.1.0). Дальше — операционные хвосты
и опциональный бэклог (`docs/status.md`).

### 2026-07-12 — переезд боевого сервера на NUC (v0.1.0)

Владелец сменил целевой хост с Raspberry Pi на NUC Minisforum
(`nuc-minisforum-wifi`, Ubuntu 26.04, x86_64) — `deploy.yml` научился
выбирать платформу образа по архитектуре цели (amd64/arm64). Прогнан
полный штатный путь на живом хосте: `make provision` (docker с нуля,
юниты) → перенос `.env` (+`EB_TIMEZONE`), `frpc.toml`, БД и VAPID-ключей
→ `make release VERSION=0.1.0` → `make deploy` — health-гейт и туннель
зелёные. Недочёты первого прогона (оба исправлены): provision включал
watchdog-таймер, но не стартовал его; профиль AppArmor `docker-default`
на Ubuntu блокировал D-Bus-вызовы контейнера к BlueZ (BLE был мёртв,
`AccessDenied ... AppArmor policy prevents`) — в compose добавлен
`apparmor=unconfined`. Ноутбук-стенд погашен; бризеры ждут повторного
сопряжения (бонды per-host). Владелец перевёл NUC с wifi на LAN —
внешний доступ на окно переключения падал, восстановился сам.

### 2026-07-12 — полевой баг: «уменьшение скорости выключает бризер»

Симптом: при перетаскивании ползунка скорости вниз бризер ~1 раз из 10
выключался. Сервер/протокол чисты (журнал `commands`: ни одна команда
`fan_speed` не вернула `power=false`) — выключения были отдельными
командами `{'power': false}` от UI. Причина: `Toggle` — кнопка-свитч
внутри `<label>` на всю ширину карточки; label пересылает клик по любому
месту строки вложенной кнопке, а строка питания лежит вплотную над
слайдером (при скоростях 5–6 ползунок — прямо под свитчем). Промах
пальцем = невидимое выключение. Фикс: обёртка `div` вместо `label`
(кликабелен только свитч) + зазор над слайдером + регрессионный тест
«клик по подписи не шлёт команду».

### 2026-07-12 — развёртывание: docker compose + ansible + версии

Решения владельца: доставка образов без registry
(`docker save | ssh | docker load`); самолечение — systemd-юнит поверх
compose + таймер health-чека (sd_notify/WatchdogSec в контейнере не
работают, restart-policy не реагирует на unhealthy); bare-systemd/venv
путь установки удалён (юнит переписан на compose, frpc на целевом хосте —
контейнером, профиль `tunnel`).

Сделано: `deploy/docker/docker-compose.yml` — прод-раскладка
`/opt/easy-breezy` (healthcheck, ротация логов, bind-mount `./data`);
`deploy/ansible/` — provision (docker, юниты, заготовки конфигов) и
deploy (buildx arm64 → tar → ssh → load, health-гейт с проверкой версии,
автооткат на предыдущую); `deploy/systemd/` — юнит compose +
watchdog-скрипт/сервис/таймер; цели `make provision|release|deploy`
(release: bump `pyproject`+`__init__`+`uv.lock`, коммит, тег `vX.Y.Z`).
Кросс-сборка arm64 проверена локально (qemu-binfmt +
buildx-builder `easy-breezy-builder` на драйвере docker-container).
**На живом Pi не гонялось** — первый прогон совмещается с переездом
стенда (`docs/status.md`, п. 2).

### 2026-07-12 — APK: версия и именование

- Найдена причина пустого `versionName` в APK: bubblewrap читает версию из
  поля `appVersion` в `twa-manifest.json`, а поле называлось
  `appVersionName` и молча игнорировалось. Поле переименовано.
- Артефакт сборки теперь `mobile/easy-breezy-<версия>.apk`; `make apk`
  переименовывает и удаляет промежуточные `app-release-*` (раньше рядом
  лежал неподписанный APK — вероятный источник «не удалось установить»).
- Диагностика установки — `mobile/README.md`.

## Итоги Фазы 9 — полировка

- **Бэкапы (NFR-7):** ежедневно 03:30 TZ `VACUUM INTO` + gzip в
  `data/backups/`, ретенция 7; восстановление проверено тестом; провал →
  событие + web push. Ручные процедуры — runbook.
- **Watchdog (NFR-4):** все бризеры офлайн > 10 мин → power-cycle адаптера
  (BlueZ D-Bus) → ещё 10 мин → SIGTERM (systemd поднимет). Юнит переведён
  на Type=notify + WatchdogSec=60 (READY/WATCHDOG через sd_notify).
- **Web push (FR-32):** VAPID-ключи в `data/vapid_private.pem`, подписка
  в Настройках PWA; уведомления: бризер офлайн > 10 мин (раз на эпизод),
  провал бэкапа. Свой SW (injectManifest: precache + push + клик).
- **Гейт покрытия:** `make test` требует ≥ 80 % (факт ~85 %).
- **APK (TWA):** обёртка PWA (обновления с сервера, web push работает;
  Capacitor отвергнут — в его WebView Push API мёртв). Подпись сверена с
  `/.well-known/assetlinks.json`. Пересборка — `make apk`
  (`mobile/README.md`).
- Runbook: установка на Pi, бэкап/восстановление, watchdog-эскалация,
  обновление версии, APK.
- Осталось владельцу: недельный soak на Pi (RSS < 300 МБ) — после переезда
  со стендового ноутбука.

## Итоги Фазы 6 — датчики CO₂ и триггеры

- Датчики: `core/sensors.py` (SensorIngest — мерж `last_values`,
  `sensor.updated`, стейл > 5 мин); MagicAir cloud (`integrations/magicair`,
  протокол сверен с референсом airens/tion, опрос 60 с, backoff до 10 мин,
  автосинк реестра) и MQTT (`integrations/mqtt`, aiomqtt: JSON-топик или
  по-метричные `{топик}/co2|temperature|humidity`, голое число = CO₂;
  недоступный брокер — реконнект, сервис живёт).
- Триггеры: `automation/triggers.py` — защёлка с гистерезисом (чистая
  `decide` + табличные тесты), кулдаун, окно HH:MM через полночь (выход
  разрешён всегда), sweep молчащих датчиков (>10 мин — warning, защёлка
  держится), ключи `trigger:{id}:{edge}:{ts}`, приоритет 1, hold →
  `skipped_hold`.
- REST `/api/sensors` + `/api/triggers`; Яндекс: климат-датчики
  (`sensor:{N}`, co2_level/temperature/humidity, стейл → UNREACHABLE,
  action → INVALID_ACTION, state/discovery callbacks).
- UI: карточки датчиков на дашборде (шкала CO₂ 800/1000 ppm), реестр
  на «Устройствах», вкладка «Триггеры», датчики на «Графиках».

## Итоги Фазы 5 — автоматизация и телеметрия

- `automation/`: `Clock` (инжектируемое время), `ScenarioService` (мерж
  per-device дельт, исполнение через шину), `SchedulerService` (croniter,
  курсор времени `Schedule.cursor_ts`, missed-fire < 5 мин исполняется,
  идемпотентный повтор после падения — дедуп журналом, пробуждение по
  `automation.changed`, вечный цикл по ADR-0007). TZ — `EB_TIMEZONE`
  (IANA-имя; по умолчанию системная, для зон с DST задавать явно).
- REST: CRUD `/api/scenarios` (+`/run` — ручной приоритет, ставит hold)
  и `/api/schedules` (валидация cron 5 полей), `GET /api/telemetry`
  (raw/hourly), `POST /api/auth/password` (сброс чужих сессий).
- UI: «Автоматика» (редактор сценариев; конструктор расписаний
  «время + дни» ↔ cron, произвольный cron — текстом), плитки сценариев
  на дашборде, «Графики» (recharts: сутки raw / 7–30 дней агрегаты),
  «Настройки» (смена пароля).
- Семантика DST зафиксирована тестами по факту croniter: весной
  несуществующее время сдвигается вперёд (один запуск), осенью
  неоднозначное срабатывает в обоих смещениях (для идемпотентных
  сценариев безвредно).
- Тесты стали герметичны от `server/.env` (стендовый
  `EB_SESSION_COOKIE_SECURE=true` ломал cookie в TestClient; `test_health`
  гонял lifespan по боевой `data_dir` — теперь `test_settings()`).
