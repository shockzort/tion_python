## Сделано (Фазы 0–1)

**Каркас (Фаза 0):** монорепо `server/` (Python 3.12, uv, FastAPI, `/api/system/health`,
structlog JSON) + `ui/` (Vite/React 19/TS/Tailwind v4/PWA) + `deploy/` (systemd, docker,
nginx, frp) + CI (GitHub Actions). `make lint` / `make test` — зелёные.

**BLE-библиотека (Фаза 1)** в `server/src/easy_breezy/ble/`:
- `protocol/framing.py` — кадрирование Lite-семейства, `Reassembler`, split, CRC-16/CCITT-FALSE
  (генерация + мягкая валидация). `protocol/s4.py` — `S4State`, decode/encode, инжектируемый nonce.
- `transport.py` — `BleTransport` + `BleakTransport` (find_device_by_address → connect по объекту,
  стадии connect/notify разнесены, таймауты). `fake.py` — `FakeS4Device`/`FakeTransport` с
  инъекцией сбоев (тесты и будущий dev-режим).
- `driver.py` — `S4Driver`: слушатель нотификаций, get/set с подтверждением кадром состояния
  и fallback-запросом, per-device сериализация.
- `supervisor.py` — `DeviceSupervisor`: реконнект с backoff 1→60 с + джиттер, опрос 30 с,
  деградация после 3 промахов, scan-gate. Время инжектируется (time-travel тесты).
- `scanner.py` — скан с фильтром имени. `cli.py` — `breezy scan/pair/unpair/state/set/monitor`.
- Тесты: 46 (golden-векторы байт-в-байт, framing, драйвер на fake, супервизор). mypy strict.

**Ключевые файлы для ориентации:** `docs/protocol/tion-s4-ble.md` (спецификация + §5 полевые
наблюдения + раздел диагностики BLE в `docs/runbook.md`), `server/tests/golden/*.json`.

## Ключевые находки сессии

### Подтверждено на живом железе (3× Tion S4 в квартире)
- Имя в эфире: **`Breezer 4S`** (без «Tion»), адрес **random-static**, реклама **ADV_IND
  (connectable)**, manufacturer data company `0xffff` = `<mac> 03 80 00 00 00`.
- **Сервисный UUID `98f00001-3788-83ea-453e-f52244709ddb` подтверждён** (в рекламе). Write/notify
  UUID (`…0002`/`…0003`) — пока не подтверждены (нужна GATT-таблица через соединение).
- Протокольный кодек не проверен на живом обмене (не дошли из-за сопряжения, см. ниже), но
  golden-векторы байт-в-байт корректны.

### Разгадка HCI 0x3E (перепроверено 2026-07-11, вторая сессия): адаптер НЕ виноват
- Симптом: скан видит все 3 бризера, но connect падает с HCI `0x3E` («Connection Failed to be
  Established» / `le-connection-abort-by-local`): `CONNECT_IND` ушёл, периферия не ответила
  ни одним пакетом, разрыв через 6 conn-интервалов. Детерминированно, 0/30+.
- **Контрольный эксперимент**: тем же Intel AX211 в той же серии подключились **3/3 чужих
  LE-устройства** (включая random-адреса, GATT прочитан); бризер — 0/2. Сбой воспроизведён на
  **простаивающем** адаптере (без A2DP/наушников/HID), WiFi на 5 ГГц.
  → Версии «WiFi/BT coexistence», «A2DP забивает радио», «неисправный модуль» **опровергнуты**.
- **Причина: Tion 4S принимает LE-соединения только от сопряжённых центральных.** Незнакомый
  `CONNECT_IND` игнорируется прошивкой молча. Новое устройство подключается только пока бризер
  в режиме сопряжения (кнопка на корпусе ~5 с до синего мигания). Так же работают
  esphome-tion/HA-tion («перед первым использованием переведите бризер в режим сопряжения»).
- Ошибка прошлой сессии: вывод §5 «сопряжение не требуется» был неверен (сделан по единичному
  необъяснённому успеху); кнопку сопряжения ни разу не нажимали. Спека §5/§1.7 исправлены.
- USB BT-донгл покупать не нужно (по крайней мере для этого блокера).

### Живой смоук: главное пройдено 2026-07-11 (вторая сессия)
- **SET-смоук**: fan/heater/temp/mode применяются; латентность ~0.15 с (цель <2 с);
  подтверждение приходит само; CRC подтверждён обоюдно; инверсия бита нагрева
  доказана физикой (out_temp 22→24 °C). Драйвер: после SET всегда контрольный
  REQUEST_PARAMS (кадр может потеряться, прошивка может скорректировать поля).
- **MagicAir — активный конкурент**: держит соединения (бризер пропадает из эфира)
  и откатывает mode/heater/temp к своей политике за 1–3 мин, меняет fan по CO₂.
  Для Фазы 2: manual-hold окна обязательны; «SET откатился» — не баг кодека.
  Владелец планирует вывести MagicAir после разработки сервиса.
- **Сопряжение выполнено** (EC:82:9F:A4:90:14): кнопка ~5 с → байт рекламы `00`→`01` →
  `Device1.Pair()` → `Paired=Bonded=true`. Ловил фоновый цикл (скрипт эпохи отладки;
  теперь это умеет `breezy pair`).
- **Бонд снимает блок**: обычный `breezy state` без кнопки прочитал состояние — тем самым
  живьём подтверждены write/notify UUID, кадрирование, CRC, декодер S4 (значения
  правдоподобны: скорость 3/6, вход 25 °C, фильтр 56 дн).
- **`breezy pair` починен** (SMP строго до GATT, BlueZ `Device1.Pair()` в транспорте;
  бризер рвёт связь, если лезть в GATT до сопряжения). `make lint`/`make test` зелёные.
  End-to-end прогон новой команды на свободном бризере — при случае (ветка AlreadyExists).
- **Наблюдение**: бризер не рекламируется, пока соединение держит другой центральный
  (приложение Tion/MagicAir) — «не найден в эфире» ≠ «выключен». Архитектурное следствие
  для супервизора и Фазы 2 задокументировано в спеке §5.

## План дальнейших действий

### Остаток hardware-чек-листа (`docs/runbook.md`, нужен человек рядом)
1. Сопрячь два оставшихся бризера: кнопка ~5 с → `breezy pair <MAC>` (D0:60:0E:F7:EA:D4,
   EB:B5:4E:13:31:B5). У дальнего RSSI −88…−91 — сопрягать с ноутбуком поближе.
2. Персистентность бонда: power-cycle бризера → `breezy state` без кнопки.
3. GATT-таблица целиком (`bluetoothctl` → `menu gatt` → `list-attributes`) → спека §1.1.
4. SET-команды (шаги 5–9): скорости на слух, **нагрев — сверка инверсии бита с приложением
   Tion**, режим/звук/подсветка; латентность и пассивный state-кадр после SET (шаги 11–12).
5. Реконнект-тест `breezy monitor` с выдёргиванием питания (шаг 10).
6. Выяснить, кто держит соединения бризеров (приложение Tion на телефоне? MagicAir с
   CO₂-автоуправлением?) — влияет на стратегию удержания соединения сервисом (Фаза 2).

### Фаза 2 — можно начинать без железа (dev-режим на FakeS4)
Ветка `feature/phase-2-core-api` от master. Объём (план §5, §8):
- `storage/` — SQLAlchemy 2.0 async + aiosqlite + Alembic; таблицы devices/groups/commands/
  scenarios/schedules/triggers/sensors/telemetry/users/sessions/oauth/settings (модель — план §5).
- `core/events.py` — event bus (in-process async pub/sub).
- `core/registry.py`, `core/state.py` — реестр устройств + кэш последнего состояния.
- `core/bus.py` — command bus: идемпотентность по ключу, per-device очереди, приоритеты, журнал,
  дедуп ретраев (план §8). `core/holds.py` — manual-override окна.
- `core/telemetry.py` — рекордер + downsampler.
- `api/rest/*` (devices, groups, commands, system) + `api/ws.py` (WS-хаб) + `api/deps.py` (auth).
- Авторизация: локальный админ (argon2id), сессии-cookie, api-токены.
- **dev-режим `EB_FAKE_DEVICES=3`** — поднять 3 фейковых бризера на `FakeS4Device`, чтобы весь
  сервис и UI Фазы 3 разрабатывались без BLE-железа.
- Тесты: дедуп/сериализация/таймауты/hold, httpx ASGI «POST command → WS event» на fake,
  alembic upgrade.

## Как возобновить

```bash
cd server && uv sync          # зависимости
make test                     # 46 тестов зелёные
make lint                     # ruff+black+isort+mypy strict, oxlint+tsc
uv run breezy scan            # найдёт бризеры (connect до сопряжения — молчаливый 0x3E, это норма)
make dev                      # сервер :8000, /api/system/health
```

Локальные ветки `feature/rewrite-phase-0`, `feature/phase-1-ble` уже влиты в master — можно
удалить. Старая `feature-search-and-register-devices` — легаси, тоже можно удалить.
