## План

### 1. APK — установка и «Опасный сайт» (шаги владельца)

Свежий `mobile/easy-breezy-1.0.0.apk` (2026-07-12): починена версия
(поле `appVersion` в twa-манифесте), артефакт именуется
`<приложение>-<версия>.apk`, промежуточные файлы сборка удаляет — раньше
рядом лежал неподписанный `app-release-unsigned-aligned.apk`, установка
которого даёт ровно «не удалось установить». Если и новый файл не встаёт:
удалить с телефона уже установленный Easy Breezy (конфликт подписи) и/или
`adb install` с ПК за точным кодом ошибки — разбор в `mobile/README.md`.

Адресная строка с «Опасный сайт» (2026-07-12): TWA-верификация в порядке
(`assetlinks.json` проверен), бровку рисует Chrome из-за вердикта
Safe Browsing на `easy-breezy.duckdns.org` — эвристический real-time флаг
(в публичной базе домена нет: «нет данных», duckdns.org чист); логин-форма
на бесплатном dyndns — типовой фишинг-паттерн. Смена эмбеддинга не лечит
(и WebView-обёртки убивают push — Фаза 9). Действия:
1) сообщить о ложном срабатывании —
   <https://safebrowsing.google.com/safebrowsing/report_error/?url=https://easy-breezy.duckdns.org/>;
2) Search Console: добавить property, «Проблемы безопасности» → если флаг
   виден — «Запросить проверку» (1–3 дня);
3) надёжно навсегда — свой домен вместо duckdns (репутация не шарится
   с фишерами); переделки малые: nginx `server_name` + certbot на VPS,
   `host` в `twa-manifest.json` + пересборка APK, `EB_PUSH_CONTACT`,
   Endpoint в консоли Яндекса — код UI/сервера домен не хардкодит.
   Гипотеза: может снять и «неизвестную ошибку» публикации навыка.
После снятия флага бровка исчезает сама; если осталась — переустановить
APK при доступном сайте (верификация кэшируется с первого запуска).

### 2. Переезд на Raspberry Pi — первый боевой прогон развёртывания

Инфраструктура развёртывания готова (2026-07-12; решения владельца:
образы `docker save | ssh | load` без registry, systemd-юнит + curl-таймер
вместо sd_notify, bare-systemd путь удалён — `docs/history.md`):
docker compose (`deploy/docker/`), ansible provision/deploy с health-гейтом
и автооткатом (`deploy/ansible/`), цели `make provision|release|deploy`.
Кросс-сборка arm64 проверена на dev-машине; **на живом Pi не гонялось**.

Шаги переезда: `inventory.yml` → `make provision` → на Pi заполнить
`.env` (обязательно `EB_TIMEZONE`) и `frpc.toml` → `make release
VERSION=0.1.0` → `make deploy VERSION=0.1.0` → сопрячь бризеры заново
(бонды per-host) → недельный soak (RSS < 300 МБ, NFR).

### 3. Фаза 8 — BLE-модуль CO2 (локально, без облака)

Владелец (2026-07-12): модули CO2+ должны добавляться как бризеры —
кнопка спаривания на модуле, мастер в UI. Протокол BLE CO2+ не
документирован, фаза начинается с разведки на реальном модуле
(нужен модуль в радиусе BLE ноутбука):

- R&D на железе: расширить фильтр сканера именами CO2+, GATT-дамп,
  проверить пейринг кнопкой (SMP-бонд как у 4S?); факты — в
  `docs/protocol/tion-co2plus-ble.md` + golden-векторы.
- Драйвер чтения (notify/опрос) поверх готового `BleTransport`;
  супервизор; измерения → `SensorIngest` третьим kind="ble".
- Мастер сопряжения различает тип устройства (бризер | датчик CO2).
- Облачный MagicAir остаётся вторым источником (антидубль по имени/зоне
  решить в фазе — вероятно, ручное отключение облачного двойника).
