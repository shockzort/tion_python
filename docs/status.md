## Отладочный стенд MVP (работает сейчас)

Внешний доступ: `https://easy-breezy.duckdns.org` → VPS `195.133.20.205`
(nginx TLS + frps, юниты systemd) → frpc на ноутбуке → сервис :8000.

- Сервис: из `server/` командой `uv run python -m easy_breezy`
  (лог — scratchpad `eb-stand.log`); данные в `server/data/easy_breezy.db`;
  секреты в `server/.env` (client-креды Яндекса, Secure-cookie).
- frpc: `~/.local/opt/frp/frpc -c ~/.local/opt/frp/frpc.toml` (токен VPS внутри).
- Учётка: `admin` (пароль выдан владельцу; смена пароля — задача Фазы 5/8).
- Бризеры (бонды на этом ноутбуке): `EC:82:9F:A4:90:14` Гостиная ·
  `D0:60:0E:F7:EA:D4` Детская · `EB:B5:4E:13:31:B5` Спальня.
  На боевом RPi сопряжение повторить (бонд per-host); у Детской/Спальни
  ресурс фильтра на нуле — владельцу заменить/сбросить.
- Пока сервис держит соединения, MagicAir/Tion Remote к бризерам не
  подключаются (ADR-0005).

## Что дала Фаза 5 (итоги)

- `automation/`: `Clock` (инжектируемое время), `ScenarioService` (мерж
  per-device дельт, исполнение через шину), `SchedulerService` (croniter,
  курсор времени `Schedule.cursor_ts`, missed-fire < 5 мин исполняется,
  идемпотентный повтор после падения — дедуп журналом, пробуждение по
  `automation.changed`, вечный цикл по ADR-0007). TZ — `EB_TIMEZONE`
  (IANA-имя; по умолчанию системная, для зон с DST задавать явно).
- REST: CRUD `/api/scenarios` (+`/run` — ручной приоритет, ставит hold)
  и `/api/schedules` (валидация cron 5 полей), `GET /api/telemetry`
  (raw/hourly), `POST /api/auth/password` (сброс чужих сессий).
- UI: «Автоматика» (редактор сценариев; конструктор расписаний
  «время + дни» ↔ cron, произвольный cron — текстом), плитки сценариев
  на дашборде, «Графики» (recharts: сутки raw / 7–30 дней агрегаты),
  «Настройки» (смена пароля).
- Семантика DST зафиксирована тестами по факту croniter: весной
  несуществующее время сдвигается вперёд (один запуск), осенью
  неоднозначное срабатывает в обоих смещениях (для идемпотентных
  сценариев безвредно).
- Тесты стали герметичны от `server/.env` (стендовый
  `EB_SESSION_COOKIE_SECURE=true` ломал cookie в TestClient; `test_health`
  гонял lifespan по боевой `data_dir` — теперь `test_settings()`).

## План дальнейшей разработки

### Фаза 6 — датчики CO₂ + триггеры (ветка `feature/phase-6-sensors`)
MagicAir cloud клиент (опрос 60 с, стейл > 5 мин — бейдж), MQTT-инжест
(aiomqtt, реконнект-цикл), реестр датчиков в UI, движок триггеров
(защёлка с гистерезисом, кулдаун, окно HH:MM; оценка на `sensor.updated`
+ минутный sweep), редактор в UI, датчики → Алиса (properties + callbacks,
когда навык оживёт); CO₂ на «Графиках». План §10, FR-22/25/26/27.

### Фаза 7 — интенты (фундамент локального голоса)
Rule-based русский парсер («включи бризер в спальне на тройку»),
`POST /api/intents/execute`, строка команд в UI; ≥40 табличных кейсов.
Поверх него позже — локальный ASR/LLM вместо Яндекса (отдельное решение).

### Фаза 8 — полировка
Web push о сбоях, бэкапы + ретенции, runbook, покрытие ≥80 %, APK по желанию.

## Как возобновить

```bash
cd server && uv sync && cd ../ui && npm install   # зависимости
make test && make lint        # 143 pytest + 18 vitest; ruff/mypy strict/oxlint/tsc
EB_FAKE_DEVICES=3 make dev    # dev на эмуляторах: :8000 (+ PWA из ui/dist)
make dev-ui                   # vite :5173 с прокси на :8000 (горячая замена)
uv run breezy state EC:82:9F:A4:90:14   # CLI по протоколу (бонд уже есть)
```

Смоук-сервис для ручной проверки поднимать на **другом порту и data_dir**
(`EB_PORT=8010 EB_DATA_DIR=/tmp/... EB_SESSION_COOKIE_SECURE=false`), гасить
по сохранённому PID — `pkill -f easy_breezy` убивает и стендовый сервис.

Стенд (реальные бризеры + внешний доступ): из `server/` —
`uv run python -m easy_breezy` в фоне; frpc — `~/.local/opt/frp/frpc -c
~/.local/opt/frp/frpc.toml`; проверка —
`curl https://easy-breezy.duckdns.org/api/system/health`.

Первый вход: `POST /api/auth/setup {setup_token, username, password}` — токен
печатается в лог при старте без пользователей; дальше login-cookie или
api-токен (`POST /api/tokens`) с `Authorization: Bearer`.

Локальные ветки `feature/rewrite-phase-0…phase-5-automation` влиты в master —
можно удалить; `feature-search-and-register-devices` — легаси, тоже удалить.
